CHAPTER 3: TEMPLATING SYSTEM

 

INTRODUCTION

 

Our templating system is the interface between our fast core and the user. The idea of the templating system is born by the desire to have a less complex interface for the end user, and also a interface that is more consistent.

 

WHAT IS THE TEMPLATING SYSTEM

 

The templating system is a simple system that works by replacing certain codes in your html file with values fetched from the database. This is comparable to the way people are used to in Word merge or other applications that uses similar systems. The advantage of our templating system is that you can edit directly the html code, and so the required knowledge of making a nice site with our system is limited to html knowledge and a basic knowledge of a webshop. We need to be familiar that IncraShop has products, categories, etc.

 

So, for our system the required knowledge is much less than for the classical Magento theme and templating. This makes our system utmost suitable for design agencies and others who are used to work with html. We replace information from IncraShop into the html code, using the special characters [and], with a code in between. These codes in the html we will call replacers, because at that location that code is replaced with information from IncraShop.

 

Sometimes we have simple things to replace, like a product name, but sometimes we have more complicated things to replace like a selection of products, or we need to replace something conditional (that is, in a certain case we need to place something on that location, and in another case we need to place something else). Our templating system provides these possibilities. Because those options are needed to make a good template (we would not be able to make a good template without them), some replacer codes can get a bit complex. The following chapters explain all about replacers and their possibilities.

 

HOW IT WORKS

 

Picture with1 <core>-> 2 <proto html> - core -> html output

 

We design can be done with prtoto html files and html static blocks. This proto html files are html files with all possible features, but codes in that proto html can be replaced on a smart way with data to make the output html files. Also it is possible to include certain html blocks to reuse certain pieces of the site to make it consistent. The html files can be combined with css and javascript as you normally would do.

 

Summary

 

The templating system works by simply replacing and processing html to a final output page. This is pretty fast as it can be sone in a single pass, combining the codes in the input files with known data in the core module.


Processing the codes is a bit more then just replacing, because there is some intelligence in the code to process conditions and loop over products and other data entities.

 

 

WHAT ARE REPLACERS

 

A replacer, from the word itself, is a simple code that we use in html code to get certain information from IncraShop and perform specific function. A replacer is recognized by open and close square bracket characters, [ ]. By these replacers, our html code will look no more than a simple html codes with no complications at all and give the same output like those of Magento or any other Webshop system. Because we use this replacer codes instead of using PHP language, or other complex programming codes, making a design is much more simple.

 

Although the idea of a replacer may seem simple, it has a lot of usage. We can use it for simple information like product names, category names, etc. and on the other hand, for far more complex information. You will find these replacers convenient and easy to use when you further understand its components.

 

 

SCOPE AND SIMPLE REPLACERS

 

\[scope.code\]

 

This replacer consists of scope and code separated with a period and enclosed with square brackets - \[scope.code\]. Scope is the table or class in the page or store and is attributed by a code which describes what value you want to retrieve or what you want the syntax to return.

 

An example replacer is the \[Product.Name\].

 

Here, Product is the scope and Name is the code and this will do retrieve the name of the product in the current store view. Replacers are easier to use because you can easily specify what you want to obtain. You can also adjust the scope according to what you need to get by specifying the object or class name.

 

Example: \[Category.Name\],

 

If you want to get the name of the category, this replacer will retrieve the name of the category in the current store view.

 

Another thing you can do with this kind of replacer is you can specify what particular values you want to get, not just the product names or category names. You can do this by defining what values you want to get in the code. Like we do in the \[Product.Name\], you can get the price of the product by \[Product.Price\]. In this way we can get all the information from IncraShop that we need. A list of all scopes and replacers is provided in the reference manual.

 

LOOPING DATA

 

\[Scope.Code=html\]

 

Sometimes the data in the webshop is more complex then just a simple replacer like the name of a product. For example if we want to get the reviews of a product, there may be more than one review, since everyone can make its own review. So the replacer \[Product.Reviews\] is not a simple replacer, it is a LOOP replacer. Loop replacers do something for each underlying data element. IN this case we want to do something for each review that there is.

The syntax for this is \[Product.Reviews= {html tags} \]. The html tags are repeated for each review. This is called the inline expression. it is also possible to use the syntax like: \[Product.reviews:blockname\]. This is the same as the inline expression, but instead of providing the html code inline, we provide a block that is repeated for each review. We call this the explicit expression.

The inline expression is much more used then the explicit expression.

 

In the part that is repeated, we can use other replacers, or even other loops again.

 

So for example we want to display all the reviews it could look something like this:

<ul>\[Product.Reviews=<li>\[ProductReview.Review\]</li>\]</ul>

 

Of course this is a bit simplified, because we have more information for a review, and want to add more html coding and styling.

 

CONDITIONALS

 

\[?condition|value1=html1|...|valueN=htmlN\]

 

Sometimes we want to display some html only in a certain case. To do this we can use conditional expressions. These have the form like this:

\[?expression|?value=html|other_value=html|other_value=html|...\]. So to indicate that it is a consition we use the question mark (?). Then we can have some possible values with the html tags that are shown when it is that value. Each value is separated with a pipe (|).

 

an example:

\[?Product.IsSale|no=price = \[Product.Price\]|yes=Now from \[product.price\] to

\[Product.SpecialPrice\]\]

 

What happens here is that we check if the product is on sale. If it is not on sale we just display the price. If it is on sale we display the old price and the new discounted price.

 

It is possible to have a default scenario in case none of the values matches. we cansimply do this by using an empty value. This can look something like this:

\[?condition|value1=html1|value2=html2|=html_default\]

The empty value at the end will be selected when the condition is not value1 and not value2.

 

MULTI CONDiTIONALS

 

\[?condition1?condition2|valueA1.valueA2=html| valueB1.valueB2=html|...\]

 

Sometimes we want to test on more than one condition. For example we want to do display something depending on the customer group or the client and also on the storeview.

We can do this with the following sample:

\[?customerGroup.Name?StoreView.name|wholeSale.english=text for english wholesale customers|Wholesale.dutch=text for duct wholesale customer|wholesale=text for other whole sale customers|english=text for other english customers|dutch=text for other ducth customers|=text for all others\]

 

So we separate the values by the dot (.). If is possible to miss some values, and the best match is selected. THat means that the match that matches both (or as many conditions as we have) will get priority. Then if less conditions match the first one in the list is mached.

 

 

REPLACERS WITH PARAMETERS

 

\[scope.code,parameters1,..,parameterN\]

 

Parameter is a quantity whose value is selected for the particular circumstances and in relation to which other variable quantities may be expressed. It is also used in replacers to define values for what the scope and code trying to retrieve. See example for further understanding.

 

Ex: \[Product.Images,3=\85(html_tags)\85\]

 

The example replacer task is to retrieve images of product and display it. But with the help of a parameter, we can limit how many images are to be displayed. You will recognize a parameter is placed after the comma and followed by an equal sign. The replacer work together with parameter by executing the html tags inside used to display images of product, and loop it according to the defined parameter, in this case, 3 times. So our output showcases 3 images of products.

 

But parameters cannot only be numbers. In some cases we can use texts, or it can also be another replacer.

 

In the list of replacers the parameters that are expected or required (some parameters can be optional) are described. Most replacers do not have parameters.

 

Some parameters are used to control the looping, or the way the value is get. Other control the formatting, for example of currencies or dates.

 

INCLUDING BLOCKS

 

\[:blockname\]

 

A blockname replacer is simple but very useful. It is used to call html file and/or include it into another html file. The syntax for this replacer contains a colon followed by the name of the block that you want to retrieve and is enclosed in a pair of square brackets.

 

Ex. \[:header\]

 

The example clearly shows what the output is. If you put this replacer in your html file, for example, in your product page, it means that you are retrieving or including a blockname called header, which is an html file, to your product page. In this case, whatever codes you have in the header file, you dont have to copy paste all of those codes, instead, just call the whole file and insert it by using this replacer. The advantage of this is that whenever you have changes in your header.html file, it automatically reflects on the other pages where you include the file in, not altering it one by one.

 

So blocks can be very useful. We can re-use our html codes!

 

TRANSLATION REPLACERS

 

[_text]

 

Traditionally, webshop comes in different languages to cater their customer needs. That is different store views in different language is very important. Translation is made easy using our templating system translation replacers.

 

To translate contents of your Webstore, you just have to put underscore at every beginning of the text or paragraph and enclosed it with a pair square bracket.

 

Example:

 

Traditional html:

<head>

<title>Newsletter</title>

</head>

 

 

This is what traditionally html for texts and content looks like.

 

<head>

<title>[_Newsletter]</title>

</head>

 

In our templating system, we added few characters in order to identify immediately which texts are subject or can be translated to different set languages.

 

The translations work with code tables. it is of course possible to use translations like [_code0001] etc, but it is much better to use the real text in a selected language that you want to translate. When the code can not be found in the translation table, the text itself is placed. SO for [_code0001] that will be "code0001".

 

 

NESTING REPLACERS

 

It is possible to nest replacers, but there are some limitations. We can nest repalcers unlimited in the html that is looped or in the html of the conditionals. So we can have a loop in a conditional in another loop which have another conditional etc.

 

it is also possible to use a replacer in a parameter, but this is limited to a single simple replacer. it can not be a conditional or a loop.

 

It is also possible to use an include block with a replacer. For example: \[:\[CmsBlock.BlockName\]\]. here we can also only use a single simple replacer.

 

Replacers in other positions are not possible.

 

 

ESCAPING

 

\] and \[.

 

Sometimes we want to use the codes [ and ] in our html or javascripts. because these codes will be replaced when using te square brackets, we need to escape them. This can be sshown like this: \] and \[.

 

 

ADVANCED CONDITIONAL BLOCK NAMING

 

\[:blockname?condition1?....?conditionN\]

 

We can add conditions to the block names. When we do this we will select a block that matches the conditions as good as possible. It works similar to the multi conditional expression.

The system will trying to locate the block with the name: blockname.value1. ....valueN.html.

If that block does not exist, it will try to find a block that has as many conditions right as possible. When there is also none, it will just find the blockname.html.

 

Example:

[:MyBlock?CustomerGroup.name?StoreView.name]

Suppose that [CustomerGroup.Name] results in "WholeSale" and [StoreView.Name] results in "English".

Then the following is done:

locate and display the block: MyBlock.WholeSale.English.html

if that block does not exist look for the blocks: MyBlock.WholeSale.html or MyBlock.English.html.

When those blocks also don't exist display the block: MyBlock.html.

 

AUTOMATIC CONDITIONAL BLOCK NAMING

 

When a block is called by the system, or by another block, it is always possible to override a block for a specific module. For example we have the header block called header.html. It is possible to override this for the product page, by defining the block: header.product.html.

 

Automatic conditional block naming is also used for design variations.

 

FORMAL DESCRIPTION

 

For those who prefer to read a more formal description. Here is how the syntax can be expressed.

Where:

(text)means optional.

(text)* means text can be repeated zero or more times.

 

 

code = ((text)(expression))*

text = all chars except [ and ], these should be escape with the sequence

/[ and /]

 

expression =

Simple replacer: \[keyword (,parameter)*\]

Loop replacer: \[keyword (,parameter)*=code\]

Translation replacer: \[_translation(,parameter)*\]

Include block replacer: \[:blockname(?keyword)*(,parameter)*\]

Conditional replacer: \[?keyword(,parameter)*|(?)condition=

code(|(?)condition=code)*\]

 

keyword = scope.replacer

scope = {customer, product, category ...}

replacer = {name, ... } depending on the scope

translation = text

blockname = name of existing block without the html extension.

parameter = code, except for translation parameters then parameter =

expression

condition = text

 

keywords are in two type: keywords that have loops, and keywords that are replaced by texts. A keyword can only be one of the two.
