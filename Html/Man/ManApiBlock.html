CHAPTER XX:REST-API

 

The REST-API is a way of communicating with IncraShop. Through the REST-API functionality we can change tables and call functions. Within this scope of the chapter there are also ways discussed to retrieve data of the tables for use of an external system and to create custom data and catch events.

With the REST-API we can easily connect other applications with IncraShop, and have two way communication. We can also write extensions for IncraShop using the REST-API. Because this is based solely on data exchange via http, we can use any language to create the extensions to IncraShop.

15.1.              SETTERS

 

In previous chapters we only spoke about the replacers as way of getting and displaying information. There is also a way in which we can set an element on a table through replacers. This is used only in certain cases, because in the normal operation Incrashop sets all the data that need to be set by itself.

 

Setters can be used as: \[SET:<replacer>,<value>\]

 

Here is <replacer> the replacer that we use to change the value of the field in the record, and the <value> is the new value that is going to be set. For example: \[Set: product.name, my new product name\] will set the name of the product to "my new product name" for the current product.

 

In the html the code for the setters is replaced by nothing, so a setter will not show in the output, unless there is an error, then the setters is replaced by the error that occurred.

 

The setting of values works on the current records like with replacers, just like the normal replacers. This means that you can replacer things in a loop, or for the current value only. With the setter functionality there is no direct way to set a value in a specific record. It that is needed consider the use of the REST-API Update functionality, or make a loop and check for the right criteria.

 

Setting is only possible for simple replacers, this means only for replacers that are not loops and that are not read only.

 

 

 

15.2.              JSON GET

 

We can get a record in JSON format for processing in our own application using the JSON get functionality. This will output the record in our html. The syntax of the JSON get is the following:

 

\[JSON:<Table_name>, GET, <id>, <Field1> <Field2> ... <Field_N>\]

 

Here is <table-name> the name of the table that we want to query. <id> is the id of the record that we want to return, and <field1> to <field_N> are the fields, identified by their replacers, that we want to return. We can specify as many replacers as we want here. It is not needed that they always have the same scope as the table we are getting, but mostly it does.

 

for example: \[JSON:Category, Get, 5, \[Category.Id\], \[Category.Name\]\]

 

This will return the category id, which should be 5, and the name of that category.

 

The result will look like this: {"Category.Id":5,"Category.Name":"sample"}

 

 

 

15.3.              JSON FILTERS

 

JSON get will only return one record that we can identify by ids ID, but sometimes we want to query all records or a subset of records. We can do this by using the JSON filter functionality.

With the JSON filter functionality we can query certain records from a table that match a filter, and set the sort order of the records. The syntax is the following:

\[JSON:<filter_name>,

(FILTER, <filter field>, <Filter value>,)*

(SORT, <sort_field>, <ascending>,)

(PARENT, <parent_table>, <Parent_id>,)

(FIELDS,) <field1>, .. , <fieldN>)\]

 

Here is <filter_name> the name of the filter. For a complete list of possible filter names see the chapter filtering, but in most cases the filtername is identical to the name of the table we want to filter.

The FILTER block can be used to specify additional filter criteria, but this block is optional. It is also possible to specify this block more then once, if we want more then one additional filter criteria on the table. The filter field, is the name of the field on which to filter, and the filter value is the value to look for.

The SORT block lets you specify the sort order of the records in the output. This block is also optional. In this block the <Sort_field> is the field on which to sort and <ascending> is either true or false depending if we want the sort order to be from low values to high values (true) or from high values to low values (false).

The PARENT block specifies the parent of this table. This is needed in case that the parent is not our current website. For example if we want to get the options of a product we need to know which product we want the options from. This can be specified by the parent block. Here is <Parent_table> the name of the table and <Parent_id> the id of the record in the parent table.

The FIELDS block will specify all the fields. This block should always be there but the FIELDS keyword is optional. <Field1> until <FieldN> are the replacers of the fields that we want to return in our json data.

 

For example: \[JSON:Product, FILTER, Name, dress, SORT, Name, false, \[Product.Id\], \[Product.Sku\], \[Product.Name\]\]

 

This will return a list of products that contain the word "dress" in the name of the product. The products are also ordered on name, and for each product the id, the SKU and the name is written in the output. The output could look like:

\[{"Product.Id":6,"Product.SKU":"GHY_437636","Product.Name","Green dress"},

{"Product.Id":4,"Product.SKU":"GHY_447346","Product.Name","Green dress"},

{"Product.Id":17,"Product.SKU":"GHY_493721","Product.Name","Yellow dress"}\]

 

Another example, if we just want to get all the ids of the products: \[JSON:Product, SORT,Id, false, \[Product.Id\]\]

This will return a sorted list of the product ids.

 

Or an example where we want to list the options of a certain product: \[JSON:ProductOption, PARENT, Product, 5, \[ProductOption.Id\], \[ProductOption,Name\]\]

This will get the name and the id of all the options of product 5.

 

Nesting filters

It is possible to get all the records of the children of a certain record. For example this will get certain products that contain the word "dress" and in the JSON output will have a array of product options for each record.

 

Example: \[JSON:Product, FILTER, Name, dress, SORT, Name, false, \[Product.Id\], \[Product.Sku\], \[Product.Name\], \[JSON:ProductOption, PARENT, Product, \[Product.Id\], \[ProductOption.Id\], \[ProductOption,Name\]\]

\]

The output of this query could be:

\[{"Product.Id":6,"Product.SKU":"GHY_437636","Product.Name","Green dress","ProductOption":[]},

{"Product.Id":4,"Product.SKU":"GHY_447346","Product.Name","Green dress","ProductOption":\[{"ProductOption.Id":1,"ProductOption.Name":"Option1"},{"ProductOption.Id":2,"ProductOption.Name":"Option2"}\]},

{"Product.Id":17,"Product.SKU":"GHY_493721","Product.Name","Yellow dress"","ProductOption":\[{"ProductOption.Id":5,"ProductOption.Name":"Option5"}\]}\ ]

 

 

 

 

15.4.              REST-API ADD

 

We can not only query values using json, we can also update, add and delete records. The format of the json data we use to add and update records is very similar to the json data returned from the filter and the get.

 

Adding a single record

 

To add a record we can use the following url:

domainname/RestApi/add/<tablename>/<replacer>/<parentTable>/<parent_id> with a post filled with post[data] = <data>.

Here is:

<tableName> the name of the table where we want to add a record. <replacer> is the replacer that we want to use to add a record. This must be a loop replacer, looping over records of the given <TableName>. <ParentTable> is the name of the parent table of the replacer, this is used in combination with the ID to locate where to insert the record. This is optional, and only have to be given if the parent is differnt from the current website.

<Parent_id> is the id of the record for which we want to insert the data.

<data> is the json data of the record, which is a object with keys that are the replacers and the values to set.

 

For example:

domainname/RestApi/Add/PromoHeaderBlock/Promoheader.Blocks/PromoHeader/1

With data: {"PromoHeaderBlock.Image":"image1.jpg","PromoHeaderBlock.Position":10,

"PromoHeaderBlock.ImageText":"This is a text with the image!",

"PromoHeaderBlock.ImageUrl":\"www.incramind.nl/page_image1.html"}

 

This will add a promoheader block to the promoheader with id = 1, and fills the block with the image, imagetext, position and url as given in the data. The other fields will have empty or default values.

 

Adding multiple records at once

It is possible to add more then one record at a time for the same table. The url is the same as the url for a single record, but only the data is not a single object but is an array or objects.

For example to add two blocks at once:

\[{"PromoHeaderBlock.Image":"image1.jpg","PromoHeaderBlock.Position":10,

"PromoHeaderBlock.ImageText":"This is a text with the image!",

"PromoHeaderBlock.ImageUrl":\"www.incramind.nl/page_image1.html"},

{"PromoHeaderBlock.Image":"image2.jpg","PromoHeaderBlock.Position":11,

"PromoHeaderBlock.ImageText":"This is a text with the second image!",

"PromoHeaderBlock.ImageUrl":\"www.incramind.nl/page_image2.html"}\]

 

Adding detailed data of underlying tables

It is possible to add data of child (underlying) tables directly, by adding it to the json data. To do this replace the value of a field of an replacer of type 'loop' with an array of records for the child table. This can be done repeatedly, and also for child tables can have data for their child tables.

For example to add also some sliders with the promoheaderblock from our previous example:

{"PromoHeaderBlock.Image":"jaap","PromoHeaderBlock.IsSlider":true,"PromoHeaderBlock.Position":10,"PromoHeaderBlock.ImageText\":\"Dit is een plaatje!","PromoHeaderBlock.ImageUrl":"www.incramind.nl/plaatje", ("PromoHeaderBlock.Sliders":\[{"PromoHeaderSlider.Text":\"Slider uno","PromoHeaderSlider.ImageUrl":\"www.image.text.uno"},{"PromoHeaderSlider.Text":"Slider dos","PromoHeaderSlider.ImageUrl":"www.image.text.dos"}\]}

 

This will add two sliders (slider uno and slider dos) to the block.

 

 

 

Checking return values

 

The add request returns also a json array with results. This is an array of objects, an object for each record that is being added, or tried to being added. The objects contain the following fields:

o   Action: (string) "Add". The action that is done.

o   Name: (string) The name of replacer that is used for the adding

o   ID (number) The new id of the record that is added. This may be 0 in case the add is not successful.

o   Result: (boolean) true/false, depending on if the record was added successfully.

o   Errors: (string) An description of the errors

 

It is possible that while trying to add multiple records there is only one resulting record. This can only happen in case that there is a general error. This error can then be found in the errors description of the result.

 

 

 

 

 

15.5.              REST-API UPDATE

 

To update existing records we can also use the rest-api. This can be done by the following url:

/domainname/RestApi/Update/tablename/record_id

And with a post filled with the json data of the record to update.

 

Here is <tableName> the name of the table for which you want to update a record and the <record_id> is the id of that record.

In the <data> there are the fields with the new values. The definition of the data is the same as for the add. (see above).

 

note: You can only update one record at the time.

 

also, you can not update the id of a record, as it is used for reference.

 

An example of the url:

/domainname/RestApi/Update/PromoHeaderBlock/2

 

with data as in the last sample

 

Modifying detailed underlying tables

 

You can also modify the underlying records in an update. But because there may be already records present (unlike the add), the functionality of handing existing records is different. We have here the following options:

 

* We are not including the loop replacer in our data

In this case the child records are not altered

 

* We give the loop replacer in our data but with an empty array

In this case all existing records are deleted

 

* We give a single object (not an array) in our data, and there is an ID field in that data with an existing id.

In this case that record is modified, and all other records are not altered.

 

* We give a single object (not an array) in our data, and there is no ID field given or the ID does not exist.

In this case that record is added, and all existing records are not altered

 

* we give an array of objects in our data

The given objects in our array are added, the existing records are first deleted!.

 

 

note the last option. In this case all the existing records are first deleted, so that the array of objects that you give will be the only objects after the update.

 

TIP: it is possible to update multiple records without given an array (which would delete all other) by repeating the replacer with a object in the list. (it is ok, if a loop replacer is included twice or more time in an object).

 

Checking return values

 

The return values are similar to the return values of the add (see above). The action will only always be "Update" and the ID the record of the id that is modified (or attempted to modify)

 

 

15.6.              REST-API DELETE

 

To delete one or multiple records we can use the following url:

 

domainname/RestApi/delete/<replacer>/<parent_table>/<parent_id>/<ids_to_delete>

Here is <replacer> the name of the loop replacer to use.

<parent_table> the name of the parent table that is the parent of the record to delete.

<Parent_id> the id of the parent record

<ids_to_delete> one id, or a comma seperated list of ids, of the records that you want to delete.

 

For example:

domainname/RestApi/delete/Promoheader.Blocks/PromoHeader/1/2,3

 

Checking return values

 

The return values are similar to the return values of the add (see above). The action will only always be "Delete" and the ID the record of the id that is removed (or attempted to remove)

 

 

15.7.              REST-API EXECUTE

 

Sometimes you want to do more things, for example update a record, add another record and delete yet another one, from the same or differnt table. Of course we can make single add, update and delete calls, but that would be more time consuming then doing it in one action, because we have to call the server everytime.

We can do this by using the execute url of the rest api. The data is a bit more complex as it is an array of actions.

 

The url to use:

domainname/RestApi/Execute

 

The data looks as follows:

(Action: Add, TableName: <tablename>, Replacer: <replacer>, parent: <parent table>, ParentId: <parent_id>, Data: <data>)*

(Action: Update, TableName: <tablename>, Id: <id>, Data: <data>)*

(Action: Delete, Replacer: <replacer>, parent: <parent table>, ParentId: <parent_id>, Id: <id>)*

 

Where:

<tablename> is the name of the table to add, or to modify

<replacer> is the loop replacer for add or delete

<parent_table> is the name of the parent table

<parent_id> is the id of the parent record

<id> is the id of the record to update or to delete

<data> is the data with field values for the add or update. This data is equivalent to the data described in the rest-api add and rest-api update sections (see above).

 

 

 

For example:

\[{\"Action\":\"Add\",\"TableName\":\"PromoHeaderBlock\",\"Replacer\":\"PromoHeader.Blocks\",\"Parent\":\"PromoHeader\",\"ParentId\":1,\"Data\":\

{\"PromoHeaderBlock.Image\":\"jaap\",\"PromoHeaderBlock.Position\":10,\"PromoHeaderBlock.ImageText\":\"Dit is een plaatje!\",\"PromoHeaderBlock.ImageUrl\":\"www.incramind.nl/plaatje\"}},\

{\"Action\":\"Update\",\"TableName\":\"PromoHeaderBlock\",\"Id\":2,\"Data\":\

{\"PromoHeaderBlock.Image\":\"jaap\",\"PromoHeaderBlock.Position\":10,\"PromoHeaderBlock.ImageText\":\"Dit is een plaatje 2!\",\"PromoHeaderBlock.ImageUrl\":\"www.incramind.nl/plaatje2\"}},\

{\"Action\":\"Delete\",\"Replacer\":\"PromoHeader.Blocks\",\"Parent\":\"PromoHeader\",\"ParentId\":1,\"Id\":3}\]

15.9.              REST-API CREDENTIALS

 

 

note: This is not yet implemented

 

Before the rest-api can be used to read or modify tables, we need to login. This can be done by the following command:

domainname/RestApi/Call/login

 

with in the post or the get the following information:

        username the back-end username

        password the none encrypted password

 

The username/password combination must match a back-end user. For this user the administrator of the back-end can decide which tables should be editable.

 

Credentials and rest-api functions

 

The same login is required for some of the rest-api call functions. The rights that are required for the user are the rights for the table(s) that are affected by the function.

 

 

15.10.        CUSTOM DATA

 

Custom data tables can be created when the extension of external application has the need for storing data. You can define custom data tables with custom field via replacers of the CustomTable scope. Once the tables are defined, it is possible to add data records to the tables and read it again. The data will be persistent.

 

Creating tables and fields

 

The tables can be created by using the following replacers:

\[CustomTable.CreateNewTable,<tablename>\]

 

This will create a new table if the <tablename> does not yet exists. This replacer returns true/false depending if the operation was successful.

 

 

\[CustomTable.DeleteTable,<tablename>\]

 

This will delete the table given by its name, and destroys all its content. This replacer returns true/false depending if the operation was successful.

 

 

\[CustomTable.AddField, <tablename>,<fieldname>,<field_description>\]

 

This will add a field to the table. Modification of tables is only allowed if there is no data in the table. This replacer returns true/false depending if the operation was successful.

 

 

\[CustomTable.DeleteField, <tablename>,<fieldname>\]

 

This will delete a field from the table.

Modification of tables is only allowed if there is no data in the table. This replacer returns true/false depending if the operation was successful.

 

 

\[CustomTable.GetNrFields, <tablename>\]

 

This will return the number of fields of the table

 

\[CustomTable.TableExists, <tablename>\]

 

This will query if a table with the given name already exists. This can be used to test if we do need to create and initialise it with data and fields, or that we have already done that before.

 

Querying tables

 

\[Website.CustomTables\]

 

loops all the custom tables.

 

\[CustomTable.Id\]

 

retrieves the ID of the current custom table.

 

\[CustomTable.Name\]

 

retrieves the name of the current custom table.

 

\[CustomTable.Description\]

 

retrieves the description of the current custom table.

 

\[CustomTable.Fields\]

 

lops over all the fields of the current custom table.

 

\[CustomTable.CreatedBy\]

 

retrieves the name of the user that created the current custom table.

 

\[CustomTable.CreatedAt\]

 

retrieves the date at which the current custom table was created.

 

\[CustomTable.LastModifiedBy\]

 

retrieves the name of the user who last modified the current custom table.

 

\[CustomTable.LastModifiedAt\]

 

retrieves the data at which the current custom table was last modified.

 

 

Querying fields

 

\[CustomField.Id\]

 

Retrives the id of the current field

 

\[CustomField.Name\]

 

Retrives the name of the current field

 

\[CustomField.Description\]

 

Retrives the description of the current field

 

\[CustomField.Flags\]

 

Retrives the flags of the current field. The flags are things like read_only etc.

 

\[CustomField.Index\]

 

Retrives the index of the current field in the custom table. This is 1 based, the first field getting index 1.

 

\[CustomField.CriteriaFrom\]

 

Retrieves the enter criteria from of the current field

 

\[CustomField.CriteriaTo\]

 

Retrieves the enter criteria to of the current field

 

 

 

Storing and retrieving data

 

 

\[CustomTable.AddRecord,<tablename>\]

 

This adds a new record to the table. The replacer returns the ID of the newly created record.

 

\[CustomTable.DeleteRecord,<tablename>,<id>\]

 

This deletes the given record from the table. This replacer returns true/false depending if the operation was successful.

 

\[CustomTable.SetData, <tablename>,<id>,<fieldname>,<value>\]

This replacer sets an individual data field. The field <fieldname> of the record with <id> in the table <tablename> will be set to the <value> given. This replacer returns true/false depending if the operation was successful.

 

\[CustomTable.GetNrRecords, <tablename>\]

 

This replacer returns the number of records in the table.

 

\[CustomTable.RecordExists, <tablename>,<id>\]

 

This replacer querries if a specified record exists.

 

\[CustomTable.GetRecords, <tablename>\]

 

This will retrieve a comma seperated list of all records in the table.

 

\[CustomTable.GetData, <tablename>,<id>,<fieldname>\]

 

This will get a single data field of a single record.

 

\[CustomTable.Records, <tablename>\]

 

This will loop all the records of a table. After this it is possible to get the data of the table by using the automatic replacers of the table, like [<tablename>.<fieldname>].

 

A small example:

15.11.        EVENTS

 

note: This is not yet implemented

 

Events are ways of calling an external system when something in the site changes or something happens. For example we might want to do something if a product is added, or when an order is placed, a product is shown, etc.

 

Creating events

 

Events can be created using the rest-api functionality with the json add, like all other tables.

The events have the following information:

 

        Event.Id

        Event.Name

        Event.Type

        Event.Url

        Event.Enabled

        Event.Blocking

        Event.CachingTime

        Event.DependingStoreView

        Event.DependingCustomer

        Event.DependingCustomerGroup

 

It is possible to create two or more events of the same type. They will be executed consequently when blocking, or parallel when none blocking.

 

List of event types
